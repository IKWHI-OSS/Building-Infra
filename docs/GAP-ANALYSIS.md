# 멀티에이전트 로직 갭 분석 (vs 한빛 3부작)

> 작성 2026-06-14. 비교 대상: 서지영 『이것이 멀티 에이전트다』 재구성 한빛+ 3부작.
> 내 로직 = `constgx/agents/`(orchestrator.py · verifier.py · tools.py · ARCHITECTURE.md).

## 0. 참고 자료 요약

**① 개념·유형** (LLM 하나로는 부족하다)
- LLM→RAG→에이전트→멀티에이전트 진화. RAG는 '고정 파이프라인(입력→조회→생성)'이라 재시도·조건변경에 약함.
- **에이전트 5요소**: 목표(Goal)·LLM(추론)·도구(Tool)·**메모리(단기/장기)**·**플래너(동적 계획수정)**.
- **멀티에이전트 5유형**: 역할분업형(순차)·계층형(목표분해·위임)·협업형(생성-평가-개선)·경쟁형(다중생성→선택)·이벤트기반형(트리거).

**② 연결·실행** (어떻게 연결되고 실행되는가)
- 상호작용 패턴: **순차 / 병렬(+결과통합) / 반복개선(+종료조건)**.
- **AgentContext**: 작업 범위 내 실행상태(목표·입력·중간추론·도구이력·메타) 동기화 계층. 장기 메모리와 구분.
- 오케스트레이터: 중앙집중 vs 분산협력(A2A) vs 혼합. 재시도·분기·예외대응.
- **MCP 5단계**: 구조화 호출 생성→클라이언트→서버 **검증(형식·도구등록·인자유효성·실행권한)**→도구실행→결과반환→재판단. FastMCP `@mcp.tool()`, stdio/SSE 전송.

**③ 운영(보안·품질·비용)** (서비스 올리기 전)
- 보안: **최소권한**(조회/수정 도구분리·고위험 노출제한), **정책검토 에이전트+룰기반 필터**(rm -rf·DROP TABLE 차단), **HITL**(민감/대규모 변경 전 승인), **감사로깅**(요청→의도→파라미터→결과 전과정), ACL·상호인증.
- 품질: **LLM-as-a-Judge**(자동 정량평가), 변경 시 **회귀평가**, Temperature/모호지시 편차→정책 구체화+Rule-based 결합으로 재현성↑.
- 비용: **컨텍스트 최소화**(선택적 주입), **캐싱**, **모델 계층화**(난도별 SLM/대형모델).

## 2. 놓친 부분 (우선순위순)

### 🔴 높음 — 이 프로젝트(파괴적 삭제 루프 포함)에 직결
1. **HITL 부재** — 삭제/대규모변경 전 사람 승인 게이트가 *코드에* 없음. 배치플랜 문서엔 "첫 배치 삭제 전 1회 확인"을 적었으나 orchestrator 미구현. 자기검증만 믿고 무인 삭제 → 위험.
2. **정책검토·룰기반 안전필터 부재** — `rm -rf`/`DROP TABLE`류 파괴적 명령 *사전* 차단 계층 없음. 내 verifier는 **사후 출력검증**이지 **사전 안전검증**이 아님.
3. **최소권한 모델 부재** — 도구 권한 개념 없음. 조회전용 vs 수정 도구 분리 없음, 고위험(삭제) 도구 노출제한 없음.
4. **AgentContext 부재(핵심 구조 갭)** — 단계 간 상태·중간결과·도구이력을 공유하는 실행메모리가 없음. 각 Step 고립 → step N 결과가 step N+1로 전달 안 됨. 기사가 가장 강조한 계층.
5. **MCP 서버 검증 단계 부재** — `tools.call` 직접 실행. MCP 5단계 핵심인 서버측 검증(형식·도구등록·인자유효성·**실행권한**)이 게이트웨이에 없음. (2단계 보류였더라도 설계에 미명시.)

### 🟡 중간 — 비전/RAG/파인튜닝 단계에서 필요
6. **동적 플래너/replan 부재** — `demo_plan`이 정적. 플래너 핵심인 "중간결과 반영 계획 동적 수정"이 없음(보정 파라미터만 합치고 계획 자체는 불변).
7. **반복개선 종료조건 미흡** — 횟수(MAX_RETRIES)만. **점수 개선율 임계치** 없음 → 보정해도 개선 없으면 같은 실패를 횟수만큼 반복.
8. **LLM-as-a-Judge 부재** — verifier가 룰기반. 결정적 검사(파일크기)엔 적합하나, **비결정적 출력**(RAG 답변·비전 결과 해석)엔 LLM 심판 평가 없음.
9. **모델 계층화 부재** — 난도별 SLM/대형모델 배치 전략 없음. **'파인튜닝 비용 최소' 제약과 직결** → 설계에 반영 가치 큼.
10. **메모리 계층(장기) 부재** — 벡터DB 장기기억 없음. RAG 기능과 통합될 지점.

### 🟢 낮음 — 포트폴리오 깊이용
11. **병렬 패턴 부재** — 순차만. 병렬 실행+결과통합(우선순위/점수) 전략 없음.
12. **유형 편중** — 계층형(목표분해)·경쟁형(다중생성→선택)·이벤트기반형(트리거) 미반영.
13. **회귀평가 하네스 부재** — 모델/프롬프트/도구 변경 시 품질 회귀 검증 없음.
14. **캐싱 계층 부재** — 반복 요청 이전결과 재활용 없음.
15. **감사추적 깊이 부족** — JSONL은 있으나 의도·컨텍스트 스냅샷까지의 '판단근거 추적'은 아님. ACL/상호인증 없음.

## 3. 권장 보강 순서

1. **AgentContext 도입**(#4) — Step 결과를 누적 공유. 이후 거의 모든 보강의 토대.
2. **HITL + 안전필터 + 최소권한**(#1·2·3) — 삭제 루프에 사람승인·파괴명령 차단·도구 권한태그(read/write/destructive). 지금 데이터 워크스트림에 바로 필요.
3. **MCP 서버 검증 단계**(#5) — 2단계 MCP 전환 시 인자/권한 검증을 서버에 내장.
4. **개선율 종료조건 + 동적 replan**(#6·7) — no-improvement 감지.
5. **LLM-as-a-Judge + 모델 계층화**(#8·9) — 비전/RAG/파인튜닝 단계. 비용최소 제약과 시너지.
6. (포트폴리오) 병렬·계층·경쟁·이벤트 유형 + 회귀평가 + 캐싱.
